# Product Requirements Document

**Product:** DS Test Prep App  
**Date:** 2026-03-08  
**Status:** Draft for review  
**Source basis:** Synthesized from the existing research reports in `docs/`

## 1. Product Summary

DS Test Prep App is a **mobile-first practice and revision platform** for undergraduate students in:

- Data Science
- Artificial Intelligence
- Computer Science
- Related computing and quantitative programs

The app is intended to help students practice course-aligned material through short, repeatable study sessions built around quizzes, concept checks, code questions, and performance feedback.

The core product direction across the docs is consistent:

- This is **not** primarily an interview-prep app.
- This is **not** a full LMS or MOOC platform.
- This **is** a structured practice layer for coursework, exams, and classroom reinforcement.
- The experience should be **mobile-first for learners** and **web-first for admin/instructor workflows**.

## 2. Vision

Build the most useful practice app for DS/AI/CS undergraduates by combining:

- fast quiz-based retrieval practice
- clear explanations and guided hints
- progress tracking by topic
- spaced revision
- optional code-based practice
- later, instructor and classroom workflows

The product should feel academically aligned, trustworthy, and operationally professional.

## 3. Problem Statement

Students currently use fragmented tools:

- coding challenge sites for programming practice
- flashcard apps for memorization
- LMS tools for coursework
- AI chat tools for explanations

These tools do not fully solve the undergraduate learning problem because they are often:

- too interview-focused
- not mapped to university course topics
- weak on concept diagnosis
- weak on structured revision
- inconsistent on mobile
- not designed around short daily practice loops

## 4. Product Goals

### Primary goals

- Help students practice course content in 5 to 15 minute sessions.
- Improve exam readiness through repeated retrieval and targeted review.
- Show learners exactly where they are weak by topic and subtopic.
- Deliver explanations and hints that teach, rather than just reveal answers.
- Create a scalable product foundation that can later support instructors, institutions, and premium features.

### Secondary goals

- Support offline access for downloaded study packs.
- Support code and SQL practice for selected subjects.
- Prepare for future classroom live-session workflows.

## 5. Non-Goals

The first product versions will not aim to be:

- a full degree-learning platform
- a video-first course platform
- a full LMS replacement
- a social network/community forum
- a broad interview-prep platform

## 6. Target Users

### Primary user

Undergraduate students in DS/AI/CS who need:

- daily practice
- exam prep
- concept revision
- code and theory reinforcement

### Secondary users

- instructors
- teaching assistants
- departments

These users become more important in later phases, not in the first core release.

## 7. Core Value Proposition

Unlike generic quiz apps or interview platforms, DS Test Prep App will provide:

- **course-aligned practice**
- **mixed question types** for theory, logic, code, and data interpretation
- **guided hints and explanations**
- **topic-level mastery tracking**
- **revision scheduling**
- **mobile-first usability**

## 8. Product Principles

- Mobile-first learner experience
- Fast start and low friction
- Short practice loops over long study sessions
- Learning support over answer dumping
- Strong content structure and quality control
- Offline tolerance where it adds real value
- Professional architecture that can expand feature by feature

## 9. Recommended Platform Strategy

### Learner mobile product

- Native mobile app built with Expo and React Native
- Primary focus on iOS and Android
- Mobile remains the main learning surface

### Learner web product

- Dedicated web app built with Next.js
- Web is a first-class product surface, not just an Expo web fallback
- Best used for browser-based study, account management, progress review, and broader access

### Admin and instructor product

- Web-based admin and instructor panels built with Next.js
- Desktop-first UX for content authoring, analytics, and classroom operations

This is now the recommended product surface split:

- **Expo/React Native** for native mobile learning experience
- **Next.js** for learner web support
- **Next.js** for admin/instructor operations
- shared APIs, shared schemas, and shared product logic where appropriate

### 9.1 Recommended State-of-the-Art Tech Stack

The recommended stack should be modern, production-ready, and strong on:

- mobile UX
- web performance
- offline support
- typed contracts
- analytics
- long-term maintainability

### 9.2 Monorepo and workspace structure

- `pnpm` workspaces
- `Turborepo` for monorepo task orchestration and caching

Recommended repo shape:

- `apps/mobile` for Expo app
- `apps/web` for learner web in Next.js
- `apps/admin` for admin/instructor web in Next.js
- `services/api` for FastAPI backend
- `packages/ui-tokens` for shared design tokens
- `packages/contracts` for shared API contracts and schemas
- `packages/config` for shared linting/typescript/config standards

### 9.3 Mobile stack

Recommended:

- Expo, on the latest stable SDK
- React Native
- Expo Router
- TypeScript
- React Native Reanimated
- React Native Gesture Handler
- Expo Secure Store for sensitive local token storage
- Expo SQLite for offline study data and cached packs

Notes:

- We should treat Expo as the native app platform, not as the main web strategy.
- Expo web can exist for internal experimentation, but the production web app should be Next.js.

### 9.4 Web stack

Recommended:

- Next.js App Router
- TypeScript
- Tailwind CSS
- Radix-based component primitives or a `shadcn/ui` style component system for web surfaces
- TanStack Query for server state
- Zustand for small client-side UI state
- React Hook Form for forms
- Zod for TypeScript-side validation

Web split:

- `apps/web` for learner-facing browser experience
- `apps/admin` for admin and instructor workflows

Important implementation principle:

- share tokens, contracts, and patterns across mobile and web
- do **not** force a brittle “single UI component system for everything” across Next.js and React Native

### 9.5 Backend stack

Recommended:

- Python 3.12+ or current stable team-supported version
- FastAPI
- Pydantic v2
- SQLAlchemy 2.0
- Alembic for migrations
- PostgreSQL as the primary relational database
- Redis for caching, rate-limiting support, short-lived coordination, and job infrastructure
- S3-compatible object storage for media, exports, and offline package assets

Job execution strategy:

- use FastAPI background tasks for light async work
- use Celery plus Redis for heavy or distributed jobs

Typical heavy jobs later:

- analytics aggregation
- content import processing
- offline package generation
- notifications
- AI retrieval/indexing pipelines

### 9.6 Realtime and offline stack

Recommended:

- WebSockets for live session updates and classroom features
- mobile sync queue with retry and idempotency
- Expo SQLite-backed local persistence for offline attempts and downloaded content

Architecture note:

- offline-first behavior is required on mobile
- web should support resilient caching, but mobile is the primary offline surface

### 9.7 Authentication and authorization

Recommended primary option:

- Clerk for managed auth across Expo and Next.js

Why this is attractive:

- mature Expo support
- mature Next.js App Router support
- secure token storage patterns for Expo
- faster implementation than building auth internally

Viable alternative:

- Supabase Auth if we want tighter auth plus database ecosystem alignment

Future requirements regardless of provider:

- role-based access control
- institution-ready SSO/SAML path
- support for student, instructor, admin, and possibly anonymous class participants

### 9.8 Observability and analytics

Recommended:

- Sentry for crash reporting, error monitoring, and tracing
- PostHog for product analytics, funnels, retention, feature flags, and behavior analysis

### 9.9 Testing stack

Recommended:

- Vitest for frontend unit and integration tests
- React Testing Library for Next.js UI tests
- React Native Testing Library for Expo app component tests
- Playwright for web end-to-end testing
- Detox for mobile end-to-end testing
- Pytest for backend testing

### 9.10 Delivery and deployment stack

Recommended:

- Vercel for Next.js apps
- EAS Build, EAS Update, and related Expo services for mobile delivery
- Dockerized FastAPI service deployment

Infrastructure principle:

- keep the API portable
- avoid coupling the backend too tightly to a single frontend hosting provider
- keep cloud choice flexible until traffic, compliance, or budget requires a hard commitment

### 9.11 Final stack recommendation

If we want the most professional modern stack for this app, the recommendation is:

- **Mobile:** Expo + React Native + Expo Router + TypeScript
- **Web:** Next.js App Router + TypeScript + Tailwind CSS
- **Admin:** Next.js App Router + Tailwind CSS + Radix-style component system
- **State/data:** TanStack Query + Zustand
- **Forms/validation:** React Hook Form + Zod on web, typed form patterns on mobile
- **Backend:** FastAPI + Pydantic v2 + SQLAlchemy 2.0 + PostgreSQL + Redis
- **Offline:** Expo Secure Store + Expo SQLite + sync engine
- **Auth:** Clerk first, Supabase Auth as an alternative
- **Observability:** Sentry + PostHog
- **Testing:** Vitest + Testing Library + Playwright + Detox + Pytest
- **Monorepo:** pnpm workspaces + Turborepo
- **Mobile delivery:** EAS
- **Web delivery:** Vercel

## 10. MVP Positioning

The first release should be a **core practice app**, not an all-in-one platform.

The MVP should answer one simple question well:

**Can a DS/AI/CS undergraduate open the app, practice by topic, get feedback, and know what to revise next?**

If the answer is yes, the MVP is on track.

## 11. Core MVP Features

These are the first features we should build because they are standard, expected, and high-value in most serious practice apps.

### 11.1 Authentication and onboarding

- email/social login
- guest mode if feasible
- onboarding by degree, semester, and subject interests
- optional diagnostic quiz

### 11.2 Subject and topic structure

- subjects such as Python, SQL, DSA, Statistics, Linear Algebra, Machine Learning basics
- topics and subtopics under each subject
- clear topic taxonomy for analytics and personalization

### 11.3 Quiz/practice engine

- MCQ
- multi-select
- true/false
- short answer
- matching or ordering if low-cost to support

Requirements:

- fast load time
- clear mobile interaction
- resume incomplete sessions
- instant submission and next-question flow

### 11.4 Explanations and feedback

- show correct/incorrect immediately
- show concise explanation
- show misconception notes when available

### 11.5 Hint ladder

- hint 1: conceptual nudge
- hint 2: guided step
- hint 3: stronger scaffold

The first versions should avoid direct answer reveals unless explicitly allowed by product policy.

### 11.6 Topic progress dashboard

- accuracy by topic
- recent activity
- weak areas
- recommended next practice

### 11.7 Spaced review / revision queue

- due-today review queue
- reappearance of missed or weak questions
- optional exam-date targeting later in this stream

### 11.8 Offline study packs

- download selected quizzes/content
- complete practice offline
- sync attempts later

## 12. Phase 2 Features

These are important, but should come after the core loop is stable.

### 12.1 Code and SQL practice

- output prediction
- fix-the-bug
- fill-in-the-blank code
- SQL result interpretation
- limited autograding

### 12.2 Better analytics

- skill map by subject and subtopic
- hint usage trends
- speed vs accuracy analysis
- confidence or mastery indicators

### 12.3 Goals and motivation

- streaks
- weekly goals
- lightweight achievements

This should remain supportive and not distort learning behavior.

## 13. Phase 3 Features

### 13.1 AI tutor

- retrieval-grounded hints
- explanation of wrong answers
- citations to trusted content
- strict anti-answer-dump behavior in assessment contexts

### 13.2 Smart study plans

- exam countdown mode
- personalized revision plans
- topic prioritization based on weakness and time remaining

### 13.3 Advanced offline support

- larger offline libraries
- smarter sync handling
- local-only study and exam simulation modes

## 14. Phase 4 Features

### 14.1 Instructor and admin workflows

- question authoring
- course pack builder
- versioning and publishing
- bulk import

### 14.2 Classroom mode

- live quizzes
- join by code or QR
- teacher pacing controls
- aggregated response view

### 14.3 Local/offline classroom delivery

- LAN-based live sessions
- hotspot/local delivery options
- later sync to cloud

This is strategically valuable, but should not delay the learner MVP.

## 15. User Stories

### Student

- As a student, I want to practice a topic in a few minutes so I can study between classes.
- As a student, I want explanations after each answer so I can understand mistakes immediately.
- As a student, I want the app to show weak topics so I know what to revise next.
- As a student, I want a daily revision queue so I do not forget older material.
- As a student, I want to study offline when connectivity is unreliable.

### Instructor

- As an instructor, I want to create topic-aligned question packs for my class.
- As an instructor, I want to run live practice sessions and view aggregated results.

### Admin

- As an admin, I want controlled publishing and versioning so course packs remain reliable.

## 16. Functional Requirements

### Learner requirements

- Users can browse subjects, topics, and quizzes.
- Users can start a quiz in under a few seconds on cached content.
- Users can answer, submit, and receive feedback with minimal friction.
- Users can pause and resume practice sessions.
- Users can view progress by topic.
- Users can access a revision queue.
- Users can download eligible content for offline use.

### Content requirements

- Questions must support metadata: subject, topic, difficulty, tags, explanation, hint ladder.
- Content versions must remain stable for active attempts.

### Analytics requirements

- Track attempts, correctness, response time, hint usage, and completion.
- Aggregate performance at topic and subtopic level.

## 17. Non-Functional Requirements

### Performance

- fast question load
- responsive tap interactions
- smooth navigation on mid-range mobile devices

### Reliability

- offline attempts must sync safely later
- retries must handle intermittent connectivity

### Accessibility

- screen-reader support
- large tap targets
- dynamic text support
- keyboard accessibility on web

### Security

- secure auth
- encrypted transport
- safe handling of learner data
- role-based access for admin workflows

### Scalability

- architecture must support later addition of AI help, code execution, and instructor operations

## 18. Content Strategy

Initial subject coverage should focus on high-demand undergraduate areas:

- Python fundamentals
- SQL and databases
- Data Structures and Algorithms
- Statistics and probability basics
- Linear algebra basics
- Machine learning foundations

Content format should include:

- concept questions
- code reasoning questions
- data interpretation questions
- exam-style mixed sets

## 19. Experience Design Requirements

### Learner UX

- one-handed mobile interaction
- short sessions
- minimal clutter
- obvious next step after every action

### Admin UX

- table and filter driven workflows
- bulk editing/import where possible
- strong validation before publishing

## 20. Monetization Direction

Recommended pricing model:

- **Free tier:** limited daily practice, basic analytics, limited offline usage
- **Student Plus:** unlimited practice, revision queue, richer analytics, offline packs
- **Pro/Premium:** AI tutor, advanced analytics, code practice extensions
- **Institution/Admin:** instructor authoring, class mode, reporting, integrations

## 21. Success Metrics

### Activation

- account created
- first quiz started
- first quiz completed

### Engagement

- weekly active learners
- quizzes completed per week
- review queue usage
- streak continuation

### Learning signals

- topic accuracy improvement
- repeat-attempt improvement
- reduced hint dependence over time

### Monetization

- free to paid conversion
- subscription retention

## 22. Risks

### Product risks

- trying to build too many surfaces too early
- weak or inconsistent content quality
- overbuilding AI before the core practice loop works

### Technical risks

- code execution complexity
- offline sync complexity
- classroom real-time infrastructure complexity

### Trust risks

- AI giving poor guidance
- unclear explanations
- weak analytics causing low perceived value

## 23. Recommended Build Order

This is the most important execution decision from the docs.

We should build **feature by feature**, starting from the standard core layer that serious study apps depend on.

### Phase 0: Foundation

- product architecture
- auth
- subject/topic data model
- question schema
- analytics event model

### Phase 1: Core learner loop

- browse topics
- start quiz
- answer questions
- see explanations
- use hint ladder
- resume session

### Phase 2: Retention and value loop

- progress dashboard
- weak-topic detection
- revision queue
- notifications/reminders if needed

### Phase 3: Premium study capabilities

- offline packs
- richer analytics
- code/SQL question support

### Phase 4: Advanced differentiation

- AI tutor
- exam planning
- study plans

### Phase 5: Instructor and institution workflows

- admin panel
- content authoring
- classroom mode
- local/offline classroom delivery

## 24. Immediate Recommendation

The first professional milestone should be:

**A polished learner MVP with topic-based quizzes, explanations, hints, progress tracking, and spaced revision.**

That milestone gives the product:

- a clear user value proposition
- measurable retention signals
- a stable platform for later premium and institutional features

## 25. Open Decisions for Review

These should be finalized before implementation begins:

- whether `apps/web` and `apps/admin` should stay separate or share one Next.js codebase
- exact initial subject set
- free-tier limits
- whether code execution is MVP or Phase 2
- whether guest mode is supported
- whether offline packs are MVP or Phase 2
- auth provider decision: Clerk vs Supabase Auth
- exact backend hosting choice

## 26. Final Recommendation

The docs point to a clear app type:

**A mobile-first undergraduate practice app for DS/AI/CS, built around quizzes, guided feedback, revision, and later expansion into code practice, AI tutoring, and classroom workflows.**

The correct execution strategy is to build it professionally in layers:

1. core learner practice loop
2. revision and analytics
3. premium study capabilities
4. AI and advanced personalization
5. instructor/admin/classroom systems

This sequencing keeps the product focused, reduces risk, and creates a strong foundation for a serious educational app.
